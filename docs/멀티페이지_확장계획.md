# 서울이도치과 — 멀티페이지(MPA) 확장 계획

> 작성: 2026-06-27 · 대상: 개발/퍼블리셔 + 의사결정용
> 결정 전제: **허브앤스포크** 구조 · **진료별 5~7개 허브페이지** · **PHP 호스팅**(이미 만든 PHP MPA 골격 위 확장)
> 현재 상태: 원페이지(16섹션) 정적 사이트 + PHP MPA 변환본 보유, 개원 전 `noindex` 초안

---

## 1. 목표

- 긴 원페이지 1개 → **목적별 페이지 다수**로 분해(홈은 요약 랜딩 유지).
- 진료별 독립 허브페이지로 **GEO/지역검색 인용력**을 끌어올림(플레이북 "진료별 GEO 허브").
- 이미 구축한 PHP 구조(`index.php` 라우터 · `lib/i18n.php` · `lang/*.php` · `t()`)를 **그대로 재사용**해 페이지만 증설.
- 4개 언어(ko/en/ja/zh) 동시 확장.

### 왜 MPA가 GEO에 유리한가 (근거)
- AI/검색은 **질의 1건 = 페이지 1개**로 매칭. "공덕 임플란트"에 답할 전용 페이지가 있어야 인용 후보가 됨.
- 페이지마다 고유 `title`·`description`·`canonical`·`JSON-LD`·FAQ → 신호 밀도↑.
- 내부 링크(허브→스포크)로 엔티티 관계가 명확해짐.

## 3. URL·라우팅 설계 (PHP)

`index.php` 라우터를 `{lang}/{page}/{sub}` 파싱으로 확장.

```
요청 URL                         → 렌더
/                                → pages/home.php        (lang=ko)
/about/                          → pages/about.php
/treatments/                     → pages/treatments/index.php
/treatments/implant/             → pages/treatments/implant.php
/en/treatments/implant/          → 동일 + lang=en
/존재하지않음/                    → 404.php (200 아님, 404 상태)
```

- 라우트 테이블(화이트리스트)로 매핑 → 임의 경로 차단(보안).
- `.htaccess`(Apache): `RewriteRule . index.php`로 모든 경로를 라우터로 위임(자산·실파일 제외). Nginx는 `try_files $uri /index.php`.
- 트레일링 슬래시 정규화(`/about` → `/about/` 301)로 canonical 일관.

## 4. PHP 아키텍처 변경

현재(최소본): `index.php`가 `index.html` 통째 렌더. MPA에선 **부분 분리 + 페이지별 본문**으로 전환.

```
index.php                 라우터: 경로·언어 판별 → 페이지 설정 로드 → 조립
config/pages.php          ★ 페이지 설정맵(라우트→템플릿·메타키·스키마타입·breadcrumb)
partials/
├── head.php              <head> 공통 + 페이지별 메타/canonical/hreflang/JSON-LD 주입
├── header.php            전역 네비(멀티페이지 메뉴 + 드롭다운: 진료과목)
└── footer.php            전역 푸터 + 사이트맵 링크 + 공통 JS
pages/
├── home.php
├── about.php  doctors.php  lab.php  experience.php  visit.php  faq.php
└── treatments/
    ├── index.php  implant.php  aesthetic.php  ortho.php  general.php  prevention.php  tmj.php
lib/i18n.php              t() · 페이지별 딕셔너리 로딩 추가
lang/                     공통 딕셔너리(nav·footer 등)
lang/pages/<page>.<lang>.php   ★ 페이지별 번역(분할 관리)
```

### 페이지 설정맵 예 (`config/pages.php`)
```php
return [
  'treatments/implant' => [
    'template' => 'treatments/implant.php',
    'dict'     => 'implant',                 // lang/pages/implant.*.php
    'title'    => 'implant.meta.title',
    'desc'     => 'implant.meta.desc',
    'schema'   => ['MedicalWebPage', 'FAQPage', 'BreadcrumbList'],
    'crumb'    => ['홈'=>'/', '진료과목'=>'/treatments/', '임플란트'=>null],
  ],
  // ...
];
```

### i18n 네임스페이스
- 공통(nav·footer·a11y)은 `lang/*.php` 유지.
- 페이지별 카피는 `lang/pages/<page>.<lang>.php`로 분리 → 한 파일 비대화 방지, 페이지 단위 번역 작업 가능.
- 키 규칙: `<page>.<block>.<elem>` 예) `implant.hero.title`, `implant.faq.q1`.
- `gen-php-lang.js` 확장: `i18n/pages/<page>.js` → `lang/pages/<page>.*.php` 컴파일.

## 5. 페이지별 콘텐츠·GEO 스펙

### 공통(모든 페이지)
- 고유 `<title>`·`meta description`·`canonical`·`og:*`.
- `hreflang` 4언어 + `x-default`(해당 페이지의 각 언어 URL).
- **BreadcrumbList** JSON-LD(신규) + 화면 빵부스러기.
- Answer-first: H1(결론) → 질문형 H2 → 표/리스트 → FAQ.

### 진료 허브 페이지(★ GEO 핵심) — 임플란트 예시 골격
1. H1: "공덕 임플란트 — 수명을 생각하는 임플란트" (결론형)
2. 첫 문단: 1~2문장 자가완결 정의(인용 가능 문장).
3. **새로운 경험 4축**: 통증을 줄이는 마취 · 붓기 케어 · 턱관절 케어 · 후속 케어.
4. 질문형 H2 + 표: 적응증/진행 단계/관리(효과·가격 단정 금지).
5. 진료별 FAQ(3~5) → 페이지 자체 `FAQPage` 스키마.
6. `MedicalWebPage`/`MedicalProcedure` 스키마, 관련 진료·기공소 내부링크.
7. (개원 후) B&A는 **심의 후** 게시 슬롯만 마련.

### 페이지별 스키마 매핑
| 페이지 | 주요 JSON-LD |
|---|---|
| `/` | Dentist(@id) · WebSite · (요약 FAQ) |
| `/about/` | AboutPage · Dentist 참조 |
| `/doctors/` | Physician/Person(개원 시 자격) |
| `/treatments/` | MedicalWebPage · ItemList(진료 목록) |
| `/treatments/*` | MedicalWebPage · MedicalProcedure · FAQPage · BreadcrumbList |
| `/lab/` | MedicalWebPage(원내기공) |
| `/visit/` | Dentist(address·telephone·openingHours, 개원 시) · Map |
| `/faq/` | FAQPage(전체) |

## 6. GEO/SEO 확장 작업

- **sitemap.xml**: 전 페이지 × 4언어 자동 생성(빌드 스크립트로). 현재 단일 → N×4.
- **llms.txt**: 페이지별 섹션·URL 추가(진료별 한 줄 요약 + 링크).
- **robots.txt**: 그대로(이미 AI 크롤러 허용).
- **내부 링크 그래프**: 홈→허브→진료 스포크, 진료↔기공소↔환자경험 상호링크, 푸터 사이트맵.
- **hreflang 매트릭스**: 페이지마다 4언어 상호참조(누락 시 색인 분리 실패).
- **dateModified**: 페이지별 유지(freshness).
- **speakable**: 페이지별 핵심 선택자 지정.

## 7. 의료광고법 가드 (페이지가 늘수록 리스크↑)

- 진료 페이지일수록 **효과·비용·전후사진** 노출 유혹↑ → **각 페이지 사전심의(제57조) 체크 필수**.
- 최상급·단정·보장 표현 금지(기존 순화 톤 유지). "무통"→"통증을 줄이는 마취" 등 기존 규칙 승계.
- B&A·치료후기·비급여 할인 = 심의 대상 → 개원·심의 후 게시 슬롯만 마련.
- 페이지 추가 시 `clinic-ad-compliance` 류 검수를 빌드 체크리스트에 포함.

## 8. 단계별 로드맵 + 작업량(CC 기준)

| 단계 | 내용 | 산출 | 상태 |
|---|---|---|---|
| **0. 골격** | partials 분리(head/header/footer), 라우터 `{lang}/{page}` 확장, `config/pages.php`, 홈 이전 | 라우팅 + 홈 렌더 | ✅ 완료 |
| **1. 1차 페이지** | about·doctors·treatments(허브)·experience·lab·visit·faq 신설(섹션 재배치), 네비 언어별 URL화 | 8페이지(홈+7) | ✅ 완료 |
| **2. 진료 허브** | implant·aesthetic·ortho·general·prevention 신설(보수적 ko 초안·FAQ·MedicalWebPage·BreadcrumbList), 허브 카드 | 5 진료 페이지 | ✅ 완료(원장 감수 전) |
| **3. GEO 마감** | hreflang 매트릭스·sitemap(52url)·llms 멀티페이지·FAQPage 스키마·BreadcrumbList·진료 드롭다운 | 색인/인용 준비 | ✅ 완료 |
| **4. 다국어** | 진료 tx.* 키 en/ja/zh 번역(88키×3) → 폴백 해소·현지화 노출 | 4언어 전 페이지 | ✅ 완료(번역 검수 권장) |
| **5. 개원 게이트** | 의료진 실명·NAP·B&A(심의 후)·`noindex`→`index`·sitemap 제출 | 공개 전환 | ⏳ 개원 시 |
| **(보류) tmj** | 턱관절·수면 별도 페이지 | config 1줄 + 페이지 1개 | 결정 대기 |

> 0~2단계 완료. 진료 페이지 ko 콘텐츠는 **보수적 정보성 초안(원장 감수 전)**, en/ja/zh는 폴백으로 ko 노출(4단계 번역). 3단계(GEO 마감)는 콘텐츠 무관하게 선제 가능.

## 9. 리스크 · 추가 결정 필요

- **콘텐츠 신규 작성량**: 진료별 페이지는 새 카피가 필요 → 의료광고법 검수 부담. (기존 원페이지 카피는 분량이 짧아 페이지 단독으로는 얇음)
- **번역량 4배**: 페이지·카피 증가분 × 4언어. 자동 생성 후 검수 필요.
- **배포 전환**: 현 라이브는 GitHub Pages(정적). PHP MPA는 **PHP 호스팅으로 이전** 필요(도메인·호스팅 확정 시점). 그 전까지는 로컬/스테이징에서 개발.
- **결정 대기**:
  - `tmj`(턱관절·수면) 별도 페이지 포함 여부 (5 vs 6 진료).
  - `facilities`(공간·장비) about 통합 vs 독립 페이지.
  - 진료 페이지 신규 카피를 누가/언제 확정(원장 감수 필요).
